# 如何添加新模板

本指南定义「添加一个新前端模板」的规范。模板 = 一套自包含的设计系统(样式 + 组件 + 布局 + 文档)。加新模板不会影响其他模板。

## 多技术栈

**不同模板可以使用不同的技术栈。** 每个 `TEMPLATE.yaml` 的 `stack` 字段声明自己的技术栈(框架/UI 库/样式方案/构建工具)。例如:
- `cloud-console`:Vue 3 + 纯手写 CSS
- `admin-element`:Vue 3 + Element Plus + UnoCSS
- 未来可能有:React + Ant Design 等

因此模板的 `src/` 结构、依赖、配置文件各不相同——这正是"完整不同的风格"模板的正常形态。选模板时,技术栈是匹配维度之一(见 registry.md)。

## 何时该加新模板

当你沉淀出一套**视觉或技术上明显不同**的前端样式(不同的配色/组件库/布局结构/交互模式),且希望未来新项目能复用它时,就值得抽成一个模板。

判断"新模板"还是"现有模板的变体":
- **技术栈不同**(换 UI 库、换 CSS 方案、甚至换框架)→ 几乎一定是新模板
- **视觉明显不同**(暗色 vs 浅色、移动端 vs 桌面、卡片流 vs 表格流)→ 新模板
- **仅主题色/字号不同,组件和布局一致** → 优先在现有模板内覆盖令牌,不新建模板

## 目录结构要求

每个模板是一个独立目录,放在 `assets/templates/<模板id>/`。**不同技术栈的模板,`src/` 内部结构可以不同**(用 EP 的会有 uno.config/vite 配置/SCSS,纯手写的没有),但以下顶层文件是所有模板共有的契约:

```
assets/templates/<模板id>/
├── TEMPLATE.yaml          # 必填:模板元信息(含 stack 技术栈声明)
├── README.md              # 必填:接入步骤(含该模板的依赖安装、配置要求)
├── src/                   # 必填:完整的源码启动包(拷进新项目能跑)
│   └── (内部结构按技术栈而定,见各模板 README)
├── config/                # 按需:构建/工具配置(vite.config / uno.config / tsconfig 等)
│                          #   纯手写 CSS 模板可能没有;EP/UnoCSS 模板必须有
└── references/            # 必填:该模板专属的参考文档
    ├── design-tokens.md   #   令牌清单(颜色/圆角/阴影/字体/间距/动效/断点)
    ├── components.md      #   公共组件/封装的 props/slots/用法
    └── layout.md          #   布局结构/栅格/响应式
```

> 模板之间**互不依赖**。一个模板的 `src/`(+ 必要的 config)拷进新项目后必须能独立运行。

## 添加步骤

1. **复制现有模板(同技术栈优先)**:把一个**技术栈相近**的模板整个复制为新目录,命名用 kebab-case,如 `dark-dashboard/`。复制而非从零写,能保证配置/依赖/构建文件完整不漏。
   - 如果新模板是全新的技术栈(无相近模板可复制),需从源项目拷贝 `package.json` / 构建配置 / 入口作为起点。

2. **改 `src/` + `config/` 的内容**:替换为新风格的样式、组件、配置。重点:
   - 设计令牌(颜色/圆角/阴影/字体)——在对应的样式文件里(SCSS 变量、CSS 变量、或 UI 库变量覆盖)
   - 组件与布局
   - 构建配置(vite/uno/tsconfig)中与样式注入、别名、按需导入相关的部分
   - 至少一个示例页,展示该模板的页面组装范式

3. **清理业务代码**:删除源项目里的业务页面、业务 API、业务图标/图片、硬编码业务文案、业务专属依赖。保留通用的:http 封装、auth、通知、布局、公共组件骨架。

4. **改 `TEMPLATE.yaml`**:
   - `id`:与目录名一致
   - `stack`:**如实声明技术栈**(framework / uiLibrary / styling / build),这是选模板的关键维度
   - `name` / `tagline`:新模板的身份
   - `useCases` / `distinctiveFeatures`:它适合做什么、视觉长什么样
   - `default`:除非要替换现有默认,否则设 `false`

5. **改 `references/`**:令牌值、组件清单、布局规则都要对应新模板的实际内容。

6. **改 `README.md`**:写清该模板的**依赖安装、配置要求、接入步骤**(不同技术栈接入方式不同,不能假设通用)。

7. **在 `registry.md` 登记**:在「模板清单」表里加一行。

## 规范要点

- **组件 API 鼓励统一(同技术栈内)**:同一技术栈的不同模板,公共组件的 props/slots 名字尽量对齐,降低认知成本。跨技术栈(如 Vue vs React)无法强求一致,各自遵循所属生态惯例即可。
- **命名空间中性化**:模板里的 storage key、事件名等用中性名(`app-auth-session`、`app:notify`),不要带业务或模板 id,避免拷进项目后还要改。
- **示例页要能跑**:模板里的示例页必须和路由配置对齐,保证拷进新项目 `npm install && npm run dev` 能直接跑起来。
- **TEMPLATE.yaml 是契约**:`stack` 和 `distinctiveFeatures` 要写得让 AI 能仅凭这段文字判断"这个模板的技术栈和风格适不适合当前需求"。
- **依赖最小化**:清理掉源项目里模板用不到的业务依赖(编辑器、PDF、特定 SDK 等),保持 `package.json` 精简。

## 自检清单(添加完新模板后)

- [ ] `src/`(+ 必要 config)拷进空项目,装好依赖后能 `npm run dev` 跑起来
- [ ] 示例页与路由配置对齐,不会启动报错
- [ ] `references/` 的令牌值与实际样式文件一致
- [ ] `TEMPLATE.yaml` 的 `id` 与目录名一致,`stack` 如实反映技术栈
- [ ] `README.md` 写清了依赖安装与配置要求
- [ ] `registry.md` 已登记
- [ ] 无业务残留:业务页面/API/图标/图片/硬编码文案/专属依赖已清理
- [ ] 无其他模板的专属内容混入(如 EP 模板里不应出现 cloud-console 的手写 CSS 类)
