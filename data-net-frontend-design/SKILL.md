# 前端设计系统模板集

一组可移植的前端设计系统模板。**不同模板使用不同技术栈**(见各模板 TEMPLATE.yaml 的 stack 字段),例如 Vue3 + 纯手写 CSS、或 Vue3 + Element Plus + UnoCSS。每个模板是一套自包含的启动包(样式 + 组件 + 布局 + 配置 + 文档),拷进新项目即可获得该风格的完整 UI。

## 触发判断(先读这一段,且必须先判定、再决定是否 invoke 本 skill)

⚠️ 调用纪律(重要):在执行下面任一判定步骤、并得出明确结论之前,
**禁止 invoke 本 skill**。本 skill 自己不会告诉自己适不适用——
必须靠下面两步先把"是不是新建项目"判清楚,判定为"是"才调用,判定为"否"就直接正常干活。
"先 invoke 再让技能说适不适用"在这里是**错误做法**,会白白污染上下文。

**核心标准:用户是不是要"从零建一个新系统"**——只是"做点前端东西"(改 bug、加功能、改样式、做个页面/组件)一律不算。

**判定步骤(按顺序,几秒内完成):**

1. **先看用户措辞**:用户有没有明确说"新建项目/从零开始/搭一个 XX 后台/搭建前端"?
   - 没有(说的是"修 bug""加功能""改样式""做个组件""做个页面"等)→ **判定结束:不触发,直接干活,禁止 invoke 本 skill**。
2. **再看工作目录**:`ls` 当前目录,看是否已有 `package.json`(且依赖了前端框架)/`src/App.*`/已有前端代码。
   - 已有前端代码 → **判定结束:不触发,正常改,禁止 invoke 本 skill**(除非用户明确说"迁移到这套设计系统")。
   - 空目录或只有后端代码,且用户要新建系统 → 判定为"是",再 invoke 本 skill。

**判定结果速查:**
- ✅ 触发(判定为"是"才 invoke):用户明确要"新建/从零做一个需要前端的系统",且目录里没有已有前端代码。典型措辞:"新建一个项目""从零开始做一个 XXX 系统""帮我搭一个 XXX 管理后台""开始做前端 / 搭建前端"。
- ❌ 不触发(直接干活,**不要 invoke**):没说要新建项目;已有前端项目里加功能/改样式/修 bug;只说"做个页面/组件";含糊的"做点界面"。

**拿不准时**:默认不 invoke,先按上面两步把信息查清楚;实在无法判定,可向用户确认一句"你是要从零新建前端项目,还是在现有项目上改?"——不要靠 invoke 本 skill 来试探。

## 触发后:先让用户选,不要直接开工(关键)

触发后,**第一步是询问用户选哪种方式开发,而不是自己挑一个模板就开干**。读 `assets/templates/registry.md` 拿到当前所有内置模板,然后向用户提供选择:

**选项清单 = 所有内置模板 + "不使用模板,发散设计"**。每个模板选项要说清:**技术栈 + 一句话风格定位**,让用户能凭描述判断要不要用。例如当前应呈现为:

> 这个管理系统需要前端,有几种开发方式,你想用哪种?
>
> 1. **云控制台模板** — Vue3 + 纯手写 CSS。浅色、#1677ff 品牌色、小圆角、轻阴影的云控制台风格。轻量,无 UI 库。
> 2. **蓝渐变管理端模板** — Vue3 + Element Plus + UnoCSS。蓝色纵向渐变底壳 + 封装的列表页三件套,组件丰富,适合数据密集型中后台。
> 3. **不使用模板,发散设计** — 我不套用任何预设模板,按你的描述自由选择技术栈和风格从头设计。
>
> (各模板的详细特征可查 `assets/templates/<id>/TEMPLATE.yaml`)

呈现时注意:
- **客观罗列,不要替用户决定**,也不要默认推荐某项(除非用户主动问"你推荐哪个")
- 描述要准(照 TEMPLATE.yaml 的 stack + tagline + distinctiveFeatures),别编造
- 用户若进一步问某个模板长什么样/区别,再去读该模板的 `references/` 给详细对比
- 用 AskUserQuestion 之类的选择交互来问最自然;也可直接文字列出让用户回数字

**用户选定后**:按所选方式进入对应分支(见下)。

### 分支 A:用户选了某内置模板
按该模板的 `README.md` 接入(不同技术栈步骤不同),后续开发遵循该模板的令牌、组件、布局规范(查其 `references/`)。

### 分支 B:用户选了"不使用模板,发散设计"
**完全释放约束**:不套用任何内置模板,本 skill 的"令牌/组件/布局/不要做"等规则**一律不再适用**。AI 自由选择技术栈(Vue/React/...)、UI 库、风格,按用户描述从零设计。仅保留一点:尊重用户的显式偏好。

### 分支 C:用户自己指定了模板/技术栈(在选项之外)
如用户直接说"就用 Element Plus 那个""我要 React+AntD":尊重并按其指定走;若匹配某内置模板则用该模板,否则发散设计。

# 选定内置模板后的开发规则(仅分支 A 适用)

> 下面所有规则(何时用什么 / 组装范式 / 落地步骤 / 写代码规则 / 不要做 / 令牌)**仅当用户选了某个内置模板时适用**。若用户选了"不使用模板,发散设计"(分支 B),这些规则一律不适用,AI 自由设计。

## 何时用什么

- **从零新建前端项目** → 先选模板(技术栈+风格) → 按该模板 `README.md` 的步骤接入(不同技术栈步骤不同)
- **加页面/组件** → 读该模板 `references/components.md` + 组装范式
- **写/调样式** → 先读该模板 `references/design-tokens.md`,用该模板的令牌机制(手写CSS 模板用 CSS 变量;EP 模板用 SCSS 变量 + EP 变量覆盖)
- **改布局/栅格/导航** → 读该模板 `references/layout.md`
- **添加新模板**(沉淀新风格/新技术栈) → 读 `assets/templates/_TEMPLATE_GUIDE.md`

## 页面组装范式

每个模板有自己的页面组装惯例,**以所选模板的 references 和示例页为准**。两个现有模板的范式:

- **cloud-console**:`PageHero → MetricCards → card/grid → DataTable`(纯组件拼装)
- **admin-element**:`BaseLayout(#search/#content) → CommonSearch + BaseTable`(EP 三件套列表页)

不要把一个模板的范式套到另一个模板上。

## 新项目落地步骤(通用框架)

1. **建项目**:按所选模板的技术栈初始化(如 Vue 用 `npm create vite`,模板 README 会给具体命令)
2. **选模板**:读 `assets/templates/registry.md`
3. **接入模板**:按该模板 `README.md` 操作——装依赖、拷 src、配构建(vite/uno 等)
4. **配菜单/路由**:按模板的路由组织方式加你的页面
5. **建页面**:按模板的组装范式写
6. 详细指引**务必读模板自己的 README**(技术栈不同,步骤差异很大)

## 写代码时的硬性规则(通用)

- **遵循所选模板的技术栈**:模板用 EP 就用 EP 组件,用纯手写 CSS 就别引入 UI 库。不在一个项目里混用不同模板的方案。
- **样式用模板的令牌机制**:别写死 hex,用该模板定义的变量/覆盖机制。
- **优先复用模板的封装组件**:模板提供的 BaseTable/CommonSearch/DataTable 等先查 `references/components.md`,别手写等价物。
- **新增组件前先查模板文档**:大概率已有封装可复用。
- **保持模板的视觉一致性**:圆角/阴影/间距遵循模板令牌,别随意加阴影或改圆角风格。

## 不要做

- ❌ 混用不同模板——一个项目只用一个模板,不把 cloud-console 的手写 CSS 类用到 EP 模板里(反之亦然)
- ❌ 引入模板技术栈之外的东西——如纯手写 CSS 模板里塞 Element Plus,或 EP 模板里又装 Ant Design
- ❌ 写死颜色/圆角/阴影——用模板的令牌/变量机制
- ❌ 手写模板已有封装的组件(表格、搜索栏、指标卡等)——先查 references
- ❌ 把菜单写死在布局里——走模板的路由/导航数据驱动机制

## 令牌速查

每个模板的令牌不同,写样式时查**所选模板**的 `references/design-tokens.md`。两个现有模板的主色巧合都是 `#1677ff`,但其余令牌(圆角/阴影/背景/字号体系)差异很大。

- cloud-console 令牌见 `assets/templates/cloud-console/references/design-tokens.md`
- admin-element 令牌见 `assets/templates/admin-element/references/design-tokens.md`
